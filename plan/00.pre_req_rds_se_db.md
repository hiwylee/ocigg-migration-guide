## 2. RDS SE 고유 제약 사항

AWS RDS Oracle SE는 일반 온프레미스 Oracle과 달리 **직접 실행 불가한 명령**이 있으며,
GoldenGate 관련 필수 설정을 RDS 전용 API로 우회해야 한다.

### 2-1. 직접 실행 불가 명령 및 대체 방법

| 일반 Oracle 명령 | RDS 대체 방법 | 비고 |
|-----------------|-------------|------|
| `ALTER SYSTEM SET enable_goldengate_replication=TRUE` | `EXEC rdsadmin.rdsadmin_util.set_configuration(...)` | 재기동 불필요 |
| `ALTER SYSTEM SET streams_pool_size=...` | RDS Parameter Group → DB 재기동 | Pending Reboot 발생 |
| `EXEC rdsadmin.rdsadmin_util.set_configuration('archivelog retention hours', N)` | 동일 API | 재기동 불필요 |
| `ALTER PROFILE DEFAULT LIMIT PASSWORD_LIFE_TIME UNLIMITED` | 제한적 지원 — 주기적 갱신 절차 수립 필요 | ⚠️ 참조: [8-3](#8-3-ggadmin-패스워드-만료-관리) |
| `ALTER SYSTEM SWITCH LOGFILE` (Redo 증설) | RDS 콘솔에서 인스턴스 파라미터 변경 | 불가 항목 다수 |

### 2-2. Oracle SE vs EE 차이 — GG 영향 항목

| 항목 | Oracle EE | Oracle SE | 대응 방안 |
|------|-----------|-----------|---------|
| GG 파라미터 자동 설정 | 일부 자동 | **수동 필수** | `rdsadmin_util` 사용 |
| Streams Pool 자동 할당 | AMM 포함 | **수동 필수** | Parameter Group 설정 |
| BITMAP Index | 지원 | **미지원** | NORMAL Index로 대체 |
| DDL 복제 범위 | 전체 DDL | TABLE만 (CREATE/ALTER/DROP/TRUNCATE) | 수동 보완 절차 수립 |
| Parallel DML GG 복제 | 지원 | 제한적 | GG Extract 파라미터 확인 |

## 3. 필수 파라미터 설정

### 3-1. `enable_goldengate_replication`

**설정 시점**: D-10 이전 (재기동 불필요 → 빠를수록 좋음)

```sql
-- [소스] 현재값 확인
SELECT NAME, VALUE FROM V$PARAMETER
WHERE NAME = 'enable_goldengate_replication';
```

```sql
-- [소스] FALSE인 경우 활성화 (RDS 전용 API)
EXEC rdsadmin.rdsadmin_util.set_configuration('enable_goldengate_replication', 'true');
```

```sql
-- [소스] 즉시 반영 확인 (재기동 불필요)
SELECT NAME, VALUE FROM V$PARAMETER
WHERE NAME = 'enable_goldengate_replication';
-- 기대값: TRUE
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| enable_goldengate_replication | TRUE | | ✅ / ❌ |

> ❌ 시 → rdsadmin_util 패키지 실행 권한 확인 (SYS 계정 또는 rdsadmin 권한 필요)

### 3-2. Archive Log 보존 기간

**설정 시점**: D-10 이전 (재기동 불필요)
**이유**: GG Extract 지연 또는 장애 시 Archive Log가 삭제되면 복구 불가

```sql
-- [소스] 현재값 확인
SELECT * FROM rdsadmin.rds_configuration
WHERE name = 'archivelog retention hours';
-- RDS 기본값: 1시간 → 운영 환경에서 절대 부족
```

```sql
-- [소스] 48시간으로 설정 (최소 24, 권장 48, GG 안정화 기간 중 72 검토)
EXEC rdsadmin.rdsadmin_util.set_configuration('archivelog retention hours', 48);
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| archivelog retention hours | ≥ 48 | | ✅ / ❌ |

> **의사결정**: 보존 기간이 길수록 RDS 스토리지 사용량 증가. DB 크기, Redo 생성 속도, GG 감시 주기를 고려하여 결정. 권장 48h.

### 3-3. Streams Pool 크기

**설정 시점**: D-10 이전 (**DB 재기동 필요 → 반드시 점검 창 확보**)
**이유**: SE는 AMM(자동 메모리 관리) 대상 외 → Streams Pool 수동 설정 없으면 GG Extract가 메모리 부족으로 ABEND

#### 필요 크기 산정

```
STREAMS_POOL_SIZE ≈ (Redo rate MB/s × 60 × 5) × 1.5
                   = Redo 5분치 분량 × 안전계수 1.5
```

```sql
-- [소스] Redo 생성 속도 확인 (최근 1시간 평균, MB/s)
SELECT ROUND(SUM(BLOCKS * BLOCK_SIZE) / 3600 / 1024 / 1024, 2) AS REDO_RATE_MB_PER_SEC
FROM V$ARCHIVED_LOG
WHERE FIRST_TIME >= SYSDATE - 1/24
  AND STANDBY_DEST = 'NO';
```

| Redo rate | 산정값 | 권장 설정값 |
|-----------|--------|-----------|
| 0.5 MB/s | 225 MB | **512 MB** (최소 기준 상향) |
| 1.0 MB/s | 450 MB | **512 MB** |
| 2.0 MB/s | 900 MB | **1 GB** |
| 5.0 MB/s | 2.25 GB | **2 GB** |

**Redo rate 실측값**: ____________ MB/s → **산정값**: ____________ MB

#### RDS Parameter Group 설정 절차

```
[AWS콘솔]
1. RDS → 파라미터 그룹 → 해당 인스턴스의 파라미터 그룹 선택
   (default 파라미터 그룹은 수정 불가 → 커스텀 파라미터 그룹 필요)
2. streams_pool_size 검색 → Byte 단위로 값 입력
   - 256MB: 268435456
   - 512MB: 536870912
   - 1GB:  1073741824
   - 2GB:  2147483648
3. 변경 사항 저장 → 인스턴스에 파라미터 그룹 연결 확인
4. RDS 인스턴스 재기동 (Pending Reboot 상태 → 점검 창에 실행)
```

> ⚠️ **AWS RDS 주의**: `ALTER SYSTEM SET streams_pool_size` 직접 실행 불가 — 반드시 Parameter Group 사용

```sql
-- [소스] 재기동 후 확인
SELECT NAME, VALUE/1024/1024 AS MB
FROM V$PARAMETER WHERE NAME = 'streams_pool_size';
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| streams_pool_size | ≥ 산정값 (최소 256MB) | MB | ✅ / ❌ |

### 3-4. Redo Log 상태 확인

**설정 시점**: D-10 (확인 후 부족 시 증설 — RDS에서 직접 증설 불가, AWS 지원 요청 또는 인스턴스 파라미터 조정)

```sql
-- [소스] Redo Log 그룹 및 크기 확인
SELECT GROUP#, MEMBERS, BYTES/1024/1024 AS MB, STATUS
FROM V$LOG
ORDER BY GROUP#;
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| Redo Log 그룹 수 | ≥ 3 | | ✅ / ⚠️ |
| 그룹당 크기 | ≥ 500 MB | MB | ✅ / ⚠️ |

> ⚠️ 기준 미달 시: Log Switch 빈도 모니터링 (`V$LOG_HISTORY`). GG Extract 실행 중 Log Switch 빈발(시간당 4회 이상)은 Extract 성능 저하 원인.

## 4. Supplemental Logging 설정

GoldenGate Extract가 Redo Log에서 변경 데이터를 정확히 추출하려면 Supplemental Logging이 필수다.
**서비스 중단 없이** 설정 가능하다.

### 4-1. DATABASE 수준 로깅 활성화

```sql
-- [소스] 현재 상태 확인
SELECT SUPPLEMENTAL_LOG_DATA_MIN  AS MIN,
       SUPPLEMENTAL_LOG_DATA_PK   AS PK,
       SUPPLEMENTAL_LOG_DATA_UI   AS UI,
       SUPPLEMENTAL_LOG_DATA_FK   AS FK,
       SUPPLEMENTAL_LOG_DATA_ALL  AS ALL_COL
FROM V$DATABASE;
```

```sql
-- [소스] Step 1: 최소 로깅 활성화 (GG 필수)
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;

-- [소스] Step 2: PK / Unique Key / FK 컬럼 로깅 (GG UPDATE/DELETE 정확도 향상)
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA (PRIMARY KEY, UNIQUE, FOREIGN KEY) COLUMNS;
```

```sql
-- [소스] 적용 확인
SELECT SUPPLEMENTAL_LOG_DATA_MIN  AS MIN,
       SUPPLEMENTAL_LOG_DATA_PK   AS PK,
       SUPPLEMENTAL_LOG_DATA_UI   AS UI,
       SUPPLEMENTAL_LOG_DATA_FK   AS FK
FROM V$DATABASE;
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| SUPPLEMENTAL_LOG_DATA_MIN | YES | | ✅ / ❌ |
| SUPPLEMENTAL_LOG_DATA_PK  | YES | | ✅ / ❌ |
| SUPPLEMENTAL_LOG_DATA_UI  | YES | | ✅ / ❌ |
| SUPPLEMENTAL_LOG_DATA_FK  | YES | | ✅ / ❌ |

### 4-2. PK 없는 테이블 — ALL COLUMNS 로깅

**이유**: PK가 없으면 GG가 UPDATE/DELETE 시 변경 대상 ROW를 특정할 수 없음 → Replicat ABEND

```sql
-- [소스] 대상 테이블 수 확인
SELECT COUNT(*) AS NO_PK_TABLE_COUNT
FROM DBA_TABLES T
WHERE T.OWNER NOT IN (
    'SYS','SYSTEM','OUTLN','DBSNMP','APPQOSSYS','AUDSYS','CTXSYS',
    'DBSFWUSER','DVSYS','EXFSYS','GGSYS','GSMADMIN_INTERNAL','GSMCATUSER',
    'GSMUSER','LBACSYS','MDSYS','OJVMSYS','OLAPSYS','ORACLE_OCM','ORDDATA',
    'ORDPLUGINS','ORDSYS','REMOTE_SCHEDULER_AGENT','SI_INFORMTN_SCHEMA',
    'SYS$UMF','SYSBACKUP','SYSDG','SYSKM','SYSRAC','WMSYS','XDB','XS$NULL',
    'GGADMIN','GGS_TEMP','MIGRATION_USER'
)
AND T.TEMPORARY = 'N'
AND T.EXTERNAL = 'NO'
AND NOT EXISTS (
    SELECT 1 FROM DBA_CONSTRAINTS C
    WHERE C.OWNER = T.OWNER AND C.TABLE_NAME = T.TABLE_NAME
    AND C.CONSTRAINT_TYPE = 'P'
);
```

```sql
-- [소스] 적용 결과 검증
SELECT
    (SELECT COUNT(*) FROM DBA_TABLES T
     WHERE T.OWNER NOT IN ('SYS','SYSTEM','OUTLN','DBSNMP','APPQOSSYS','AUDSYS',
         'CTXSYS','GGSYS','LBACSYS','MDSYS','ORDDATA','ORDSYS','WMSYS','XDB')
     AND TEMPORARY='N' AND EXTERNAL='NO'
     AND NOT EXISTS (SELECT 1 FROM DBA_CONSTRAINTS C
         WHERE C.OWNER=T.OWNER AND C.TABLE_NAME=T.TABLE_NAME
         AND C.CONSTRAINT_TYPE='P')
    ) AS NO_PK_TABLES,
    (SELECT COUNT(DISTINCT OWNER || '.' || LOG_GROUP_TABLE)
     FROM DBA_LOG_GROUPS
     WHERE LOG_GROUP_TYPE = 'USER LOG GROUP'
    ) AS SUPLOG_APPLIED
FROM DUAL;
-- 기대값: NO_PK_TABLES = SUPLOG_APPLIED
```

|------|--------|--------|------|
| NO_PK_TABLES | — | | — |
| SUPLOG_APPLIED | = NO_PK_TABLES | | ✅ / ❌ |

### 4-3. 파티션 테이블 Supplemental Log 확인

```sql
-- [소스] 파티션 테이블 중 Supplemental Log 미적용 목록
SELECT PT.OWNER, PT.TABLE_NAME,
       CASE WHEN EXISTS (
           SELECT 1 FROM DBA_LOG_GROUPS LG
           WHERE LG.OWNER = PT.OWNER
           AND LG.LOG_GROUP_TABLE = PT.TABLE_NAME
       ) THEN 'OK' ELSE 'MISSING' END AS SUPLOG_STATUS
FROM DBA_PART_TABLES PT
WHERE PT.OWNER NOT IN (
    'SYS','SYSTEM','OUTLN','DBSNMP','APPQOSSYS','AUDSYS','CTXSYS',
    'MDSYS','OJVMSYS','OLAPSYS','ORDDATA','ORDSYS','WMSYS','XDB'
)
ORDER BY SUPLOG_STATUS DESC, PT.OWNER, PT.TABLE_NAME;
```

| 항목 | 기대값 | 실제값 | 판정 |
|------|--------|--------|------|
| SUPLOG_STATUS = 'MISSING' 건수 | 0 | | ✅ / ❌ |

> MISSING 발견 시 → PK 보유 여부 확인 후: PK 없으면 [4-2]와 동일하게 ALL COLUMNS 로깅 추가

## 5. GGADMIN 계정 생성 및 권한

### 5-1. 계정 생성

```sql
-- [소스] GGADMIN 계정 생성
CREATE USER GGADMIN IDENTIFIED BY "<strong_password>"
    DEFAULT TABLESPACE USERS
    TEMPORARY TABLESPACE TEMP
    PROFILE DEFAULT;
```

### 5-2. 필수 권한 부여

```sql
-- [소스] 시스템 권한
GRANT CREATE SESSION        TO GGADMIN;
GRANT ALTER SESSION         TO GGADMIN;
GRANT SELECT ANY DICTIONARY TO GGADMIN;
GRANT SELECT ANY TABLE      TO GGADMIN;
GRANT FLASHBACK ANY TABLE   TO GGADMIN;
GRANT EXECUTE ON DBMS_FLASHBACK TO GGADMIN;

-- [소스] GG LogMiner 관련 V$ 뷰 권한
GRANT SELECT ON V_$DATABASE         TO GGADMIN;
GRANT SELECT ON V_$LOG              TO GGADMIN;
GRANT SELECT ON V_$LOGFILE          TO GGADMIN;
GRANT SELECT ON V_$ARCHIVED_LOG     TO GGADMIN;
GRANT SELECT ON V_$LOG_HISTORY      TO GGADMIN;
GRANT SELECT ON V_$TRANSACTION      TO GGADMIN;
GRANT SELECT ON V_$SESSION          TO GGADMIN;
GRANT SELECT ON V_$PARAMETER        TO GGADMIN;
```

### 5-3. 권한 확인

```sql
-- [소스] 부여된 권한 확인
SELECT PRIVILEGE FROM DBA_SYS_PRIVS WHERE GRANTEE = 'GGADMIN'
UNION ALL
SELECT 'ROLE: ' || GRANTED_ROLE FROM DBA_ROLE_PRIVS WHERE GRANTEE = 'GGADMIN'
ORDER BY 1;
```

### 5-4. 접속 테스트


## 7. 사전 인벤토리 — GG 영향 오브젝트 파악

Extract 구성 전 아래 오브젝트 현황을 파악해야 한다. 발견 시 의사결정([8절](#8-의사결정-필요-항목))이 필요하다.

### 7-1. BITMAP Index

```sql
-- SE 타겟에서 BITMAP Index 미지원 → 발견 시 대체 방안 수립 필수
SELECT OWNER, INDEX_NAME, TABLE_NAME, INDEX_TYPE
FROM DBA_INDEXES
WHERE INDEX_TYPE = 'BITMAP'
AND OWNER NOT IN (
    'SYS','SYSTEM','OUTLN','DBSNMP','APPQOSSYS','AUDSYS','CTXSYS',
    'MDSYS','OJVMSYS','OLAPSYS','ORDDATA','ORDSYS','WMSYS','XDB'
);
```

| 항목 | 건수 | 비고 |
|------|-----|------|

### 7-2. XMLTYPE 컬럼

```sql
-- GG는 XMLTYPE 복제 미지원 (또는 제한적) → 복제 제외 여부 결정 필요
SELECT OWNER, TABLE_NAME, COLUMN_NAME, DATA_TYPE
FROM DBA_TAB_COLUMNS
WHERE DATA_TYPE IN ('XMLTYPE', 'SYS.XMLTYPE')
```

#### Extract 파라미터 권장 설정 (LOGALLSUPCOLS / UPDATERECORDFORMAT COMPACT)

Extract 파라미터 파일에 아래 두 옵션을 반드시 포함해야 합니다.

```
LOGALLSUPCOLS
UPDATERECORDFORMAT COMPACT
```

| 파라미터 | 역할 | 필요 이유 |
|----------|------|-----------|
| `LOGALLSUPCOLS` | UPDATE 발생 시 Supplemental Log 컬럼의 **before image 전체**를 Trail에 기록 | PK 없는 테이블에서 Replicat 충돌 방지. `HANDLECOLLISIONS` 사용 시 필수 |
| `UPDATERECORDFORMAT COMPACT` | UPDATE Trail 레코드를 **변경된 컬럼만 포함**하는 compact 형식으로 기록 (기본값: full) | Trail 크기 절감 및 AWS→OCI 광역 구간 네트워크 부하 감소 |

> **조합 효과**: `LOGALLSUPCOLS`로 before image는 전체 유지, `UPDATERECORDFORMAT COMPACT`로 after image는 변경분만 기록 → 정합성과 효율성 동시 확보

#### GoldenGate 내부 오브젝트 TABLEEXCLUDE (Checkpoint 파일 제외)

GoldenGate가 자동 생성하는 checkpoint 테이블 및 내부 스키마는 **반드시** 복제 대상에서 제외해야 합니다. 제외하지 않으면 Replicat에서 순환 복제 또는 충돌이 발생할 수 있습니다.

```
-- GoldenGate 자체 스키마
TABLEEXCLUDE GGADMIN.*;
TABLEEXCLUDE GGS_TEMP.*;

-- OCI GG Managed Service 내부 오브젝트
-- (OCI GG가 타겟 DB에 생성하는 checkpoint 스키마)
TABLEEXCLUDE C##GGADMIN.*;
TABLEEXCLUDE GGTEST.*;

-- Oracle 시스템 스키마 (GGSYS 포함)
TABLEEXCLUDE GGSYS.*;
TABLEEXCLUDE SYS.*;
TABLEEXCLUDE SYSTEM.*;
```

> ⚠️ OCI GG Managed Service는 소스/타겟 DB에 `C##GGADMIN` 또는 `GGTEST` 스키마로 checkpoint 오브젝트를 생성할 수 있음. 실제 배포 환경에서 생성된 스키마명을 `INFO ALL` 또는 `DBLOGIN` 후 `LIST TABLES` 로 확인 후 추가 제외 필요.

## 9. 운영 영향 및 주의사항

### 9-1. 서비스 중단 없이 가능한 작업

아래 설정은 **운영 중 적용 가능** (서비스 중단 불필요):

| 작업 | 영향 |
|------|------|
| `enable_goldengate_replication = TRUE` | 즉시 적용, Redo 생성량 소폭 증가 |
| `archivelog retention hours` 변경 | 즉시 적용, 스토리지 사용량 증가 |
| `ALTER DATABASE ADD SUPPLEMENTAL LOG DATA` | Online DDL, Redo 생성량 소폭 증가 |
| `ALTER TABLE ... ADD SUPPLEMENTAL LOG DATA (ALL) COLUMNS` | Online DDL, Redo 생성량 증가 |
| GGADMIN 계정 생성 | 즉시 |
| 보안그룹 인바운드 규칙 추가 | 즉시 |

### 9-2. 재기동이 필요한 작업

| 작업 | 재기동 범위 | 점검 창 필요 |
|------|-----------|-----------|
| `streams_pool_size` 변경 (RDS Parameter Group) | RDS 인스턴스 재기동 (Pending Reboot) | **필요** |
| Redo Log 그룹/크기 변경 | RDS 지원 케이스 필요 | **필요** |

> ⚠️ **RDS Multi-AZ 구성인 경우**: Pending Reboot 시 Failover 발생. 재기동 시간(통상 1~2분) 내 애플리케이션 연결 끊김 발생 가능 → 애플리케이션팀 사전 고지 및 재연결 로직 확인

### 9-3. Redo 생성량 증가 예상

Supplemental Logging 활성화 후 Redo 생성량이 약 **10~30% 증가**할 수 있다.

| 상황 | Redo 증가 추정 |
|------|-------------|
| MIN Supplemental Log만 | 5~10% |
| PK/UK/FK 추가 | +5~10% |
| PK 없는 테이블 ALL COLUMNS | 테이블 규모에 따라 가변 (최대 +50%) |

→ streams_pool_size 산정 시 활성화 **이후** Redo rate 재측정 권장

### 9-4. GG Extract 실행 중 운영 주의사항

| 상황 | 영향 | 대응 |
|------|------|------|
| GGADMIN 패스워드 만료 | Extract 즉시 ABEND | 사전 갱신 절차 수립 |
| archivelog 조기 삭제 | Extract 재시작 불가 | 보존 기간 충분히 설정 |
| RDS 인스턴스 재기동(패치 등) | Extract 연결 끊김 → 자동 재연결 확인 | GG AUTORESTART 파라미터 설정 |
| DDL 변경 (복제 범위 외) | 타겟과 스키마 불일치 | DDL 동결 정책 또는 수동 동기화 절차 |
| 신규 테이블 생성 | GG 자동 포함 여부 확인 | `TABLE <schema>.*;` 와일드카드 사용 시 자동 포함, Supplemental Log는 수동 추가 필요 |
